# 2. 函数中的所有权

> 类型：**Study note**
> 关键词：function boundary、move、return ownership、Copy
> 上一篇：[1. 所有权基础](./1-所有权.md)
> 下一篇：[3. 借用基础](./3-借用基础.md)

## 一分钟结论

- 函数调用会形成新的**所有权边界**
- 参数如果按值传入，可能会发生 move
- 返回值会把所有权带回调用方
- `Copy` 类型经过函数调用后通常仍可在调用方继续使用

## 证据来源

- 对应模块：[ownership_in_functions.rs](../../chapters/chapter02/src/ownership_in_functions.rs)
- 运行章节：`cargo run -p chapter02`

关键输出：

```text
function took ownership of vec: [1, 2, 3]
clone 后，调用方仍可继续使用 vec_1: [1, 2, 3]
function gave ownership back: [4, 5, 6]
after take and return, new owner is vec_3: [4, 5, 6, 10]
Copy type after function call still available: 10
```

## 定义

函数边界下的所有权，指的是：

- 值进入函数时，谁拥有它
- 值离开函数时，谁继续拥有它

这是所有权规则在“调用关系”上的具体体现。

## 作用

它决定了：

- 调用函数之后原值还能不能继续用
- 这个函数是在“消费值”还是“临时使用值”
- API 是否会让调用方承担额外复制成本

## 原理

### 1. 按值传参可能会 move

```rust
fn takes_ownership(vec: Vec<i32>) {
    println!("{:?}", vec);
}
```

如果调用：

```rust
takes_ownership(vec_1);
```

那么：

- `vec_1` 的所有权进入函数
- 函数结束后，`vec_1` 在调用方就不再可用

### 2. 返回值会把所有权带回来

```rust
fn gives_ownership() -> Vec<i32> {
    vec![4, 5, 6]
}
```

调用方接住返回值后，就成为新的所有者。

### 3. 函数可以“拿走后再还回来”

```rust
fn takes_and_gives_ownership(mut vec: Vec<i32>) -> Vec<i32> {
    vec.push(10);
    vec
}
```

这表示：

- 函数先获得所有权
- 在内部修改
- 再通过返回值还给调用方

### 4. `Copy` 类型跨函数边界时更轻

```rust
fn stack_function(value: i32) {
    println!("{value}");
}
```

调用后原值仍可继续使用，因为 `i32` 是 `Copy`。

## 最小示例

```rust
fn takes_ownership(vec: Vec<i32>) {
    println!("{:?}", vec);
}

fn gives_ownership() -> Vec<i32> {
    vec![4, 5, 6]
}

fn main() {
    let vec_1 = vec![1, 2, 3];
    takes_ownership(vec_1.clone());
    println!("{:?}", vec_1);

    let vec_2 = gives_ownership();
    println!("{:?}", vec_2);
}
```

## 注意点

### 1. 函数签名本身就在表达所有权语义

- `fn f(x: T)`：可能拿走值
- `fn f(x: &T)`：只借用
- `fn f(x: &mut T)`：可变借用

### 2. 为了“保留原值”而到处 `clone`，通常不是长久解法

更好的问题是：

- 这个函数到底应该拿走值，还是只借用？

### 3. 返回值有时是“归还所有权”，有时是“生成新值”

这两种 API 设计语义不同，要分开理解。

## 常见错误

### ❌ 错误 1：把函数调用当成“只是进去看看”

如果签名按值接收，它可能真的会拿走值。

### ❌ 错误 2：move 之后还用旧变量名

这是第 1 节在函数场景里的延伸表现。

### ❌ 错误 3：见到所有权问题就用返回值“绕回来”

有时更合理的做法是借用，而不是绕一圈把值还回来。

## 我的理解

- 函数边界让所有权问题第一次真正变得“工程化”
- 变量赋值只是局部现象，函数签名才是长期 API 语义
- 读 Rust 函数，第一眼就该先看参数是值、不可变引用还是可变引用
