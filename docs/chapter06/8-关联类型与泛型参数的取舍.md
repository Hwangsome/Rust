# 8. 关联类型与泛型参数的取舍

## 定义

这一节不是介绍新语法，而是比较两个抽象工具：

- 关联类型：每个实现固定一种内部类型关系
- 泛型参数：允许同一个类型针对不同类型参数写多套实现

## 作用

- 帮助你在设计 trait 时选对抽象粒度
- 避免 API 过重或过死
- 提高实现和调用两边的可读性

## 原理

像 `Bag` 这种集合，“元素类型”通常只有一种，适合关联类型。像 `Score` 这种“我既能和另一个 Score 组合，也能和一个 `u32` 组合”，更适合写成带泛型参数的 trait。

## 最小示例

```rust
trait PeekableCollection {
    type Item;
    fn first(&self) -> Option<Self::Item>;
}

trait Combine<Rhs> {
    type Output;
    fn combine(self, rhs: Rhs) -> Self::Output;
}
```

## 注意点

- “一个实现只有一种答案”时优先考虑关联类型
- “同一个左值类型要对多种右值类型实现行为”时优先考虑泛型参数
- 选型核心不是语法喜好，而是语义稳定性

## 常见错误

- 什么都用泛型参数，导致签名噪音很大
- 什么都用关联类型，结果失去多套实现能力
- 先选语法，后想语义

## 我的理解

这个取舍题本质上是在问：你要表达的是“固定关系”，还是“可变组合”。想清楚这个问题，设计 trait 时就不容易摇摆。
