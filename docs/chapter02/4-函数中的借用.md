# 4. 函数中的借用

> 类型：**Study note**
> 关键词：`&T`、`&mut T`、API 设计、slice
> 上一篇：[3. 借用基础](./3-借用基础.md)
> 下一篇：[5. 解引用](./5-解引用.md)

## 一分钟结论

- 函数签名里的借用，决定了这个函数到底是：
  - 只读使用
  - 还是允许修改调用方的数据
- `&T` 适合只读访问
- `&mut T` 适合原地修改
- 借用签名通常比“按值接收再返回”更自然

## 证据来源

- 对应模块：[borrowing_in_functions.rs](../../chapters/chapter02/src/borrowing_in_functions.rs)
- 运行章节：`cargo run -p chapter02`

关键输出：

```text
immutably borrowed vec = [1, 2, 3]
vec_1 after mutable borrow = [1, 2, 3, 10]
Math before update: [85, 90, 85]
Math after update: [85, 90, 85, 99]
```

## 定义

函数中的借用，就是把引用作为参数传入函数，而不是把所有权转移进去。

最常见两种签名：

```rust
fn read_only(x: &T) { ... }
fn update(x: &mut T) { ... }
```

## 作用

这样做的收益非常直接：

- 避免不必要的 move
- 避免不必要的 clone
- 让函数语义更清楚
- 让调用方保留对值的控制权

## 原理

### 1. `&T` 表达“只读接口”

```rust
fn borrows_vec(vec: &[i32]) {
    println!("{:?}", vec);
}
```

当前例子里还特地用了 `&[i32]` 而不是 `&Vec<i32>`，这是个很好的 API 设计信号：

- 只要能按切片读，就没必要把接口绑死在 `Vec`

### 2. `&mut T` 表达“允许原地修改”

```rust
fn mutably_borrows_vec(vec: &mut Vec<i32>) {
    vec.push(10);
}
```

这里调用方仍然拥有 `vec_1`，但函数被授予了“暂时修改它”的权限。

### 3. 函数签名其实是在设计契约

调用者看到签名，就能提前知道：

- 这个函数会不会拿走值
- 会不会修改值
- 我调用完以后原值还能不能继续用

## 最小示例

```rust
fn borrows_vec(vec: &[i32]) {
    println!("{:?}", vec);
}

fn mutably_borrows_vec(vec: &mut Vec<i32>) {
    vec.push(10);
}

fn main() {
    let mut vec_1 = vec![1, 2, 3];
    borrows_vec(&vec_1);
    mutably_borrows_vec(&mut vec_1);
    println!("{:?}", vec_1);
}
```

## 注意点

### 1. 只读接口尽量收窄

如果函数只需要读，不要为了图省事就直接写 `&mut T`。

### 2. `&[T]` 往往比 `&Vec<T>` 更通用

当前代码里这个细节很值得记住，因为它体现了“接口面向能力，而不是面向具体容器”的设计思路。

### 3. 借用规则在函数边界外仍然有效

函数只是借用发生的地方之一，不会因为“进了函数”就绕过规则。

## 常见错误

### ❌ 错误 1：明明只读，却写成按值接收

这会让调用方平白承担 move 或 clone 成本。

### ❌ 错误 2：明明不修改，却写成 `&mut`

这会不必要地提高调用约束。

### ❌ 错误 3：借用生命周期还没结束，就想发起另一种冲突借用

函数调用并不会魔法般绕开借用规则。

## 我的理解

- 函数签名里的引用，本质上是在声明“访问权限”
- 好的 Rust API 往往一眼就能从签名看出资源语义
- 写函数时先问一句“我要拥有这个值，还是只需要访问它”，会省很多返工
