`topic_05_reference_cycles.rs` 文件头用 ASCII 图解释**引用环**：`A.next → B`、`B.prev → A` 两边都强引用 → strong_count 永不归 0 → 内存泄漏。

- 解法：一侧改用 `Weak<T>` —— 只增加 `weak_count`，**不**保证底层仍活
- 访问时用 `weak.upgrade() -> Option<Rc<T>>` 动态探测
- 本节最小例子打印 `branch.strong_count` / `weak_count`

```text
branch strong = 1, weak = 1
leaf parent upgrade => Some(5)
branch children => [3]
```

## 定义

引用环是指多个引用计数指针互相形成闭环，导致它们的强引用计数永远无法降到 0。

## 作用

- 这一节的目标不是“使用引用环”，而是识别和避免它
- 帮助理解为什么 `Weak` 是 Rc 模型的重要补充
- 给树、图、父子关系建模提供一个基本准则

## 原理

父节点拥有子节点通常合理，所以用 `Rc`。子节点只需要“回看父节点”，不该延长父节点寿命，所以更合理的选择是 `Weak`。这样父节点一旦没有强引用，就仍然可以被释放。

## 最小示例

```rust
struct Node {
    value: i32,
    parent: RefCell<Weak<Node>>,
    children: RefCell<Vec<Rc<Node>>>,
}
```

## 注意点

- `Weak` 不能直接使用，通常要先 `upgrade()`
- `Weak` 不计入 strong count
- 只要结构里存在双向关系，就要主动问“哪一边才是真正 owner”

## 常见错误

- 图方便，父子双方都用 Rc
- 不区分“拥有关系”和“可访问关系”
- 看不懂 strong count / weak count 的意义

## 我的理解

引用环不是某个库的坑，而是建模问题。只要把“谁真正拥有谁”画清楚，`Weak` 的位置通常就能自然出现。
