## 定义

这一节在双向链表上补齐头删和遍历，让结构从“能搭起来”进一步走到“能维护状态”。

## 作用

- 理解双向结构删除节点时需要同步维护的状态
- 练习在 Rc / RefCell 环境下安全移动节点引用
- 看清双向结构为何比单向结构更重

## 原理

删除头节点时，不仅要把 `head` 指向下一个节点，还要把新头的 `prev` 置空。如果链表只剩一个节点，还要同时把 `tail` 也清空。双向结构的难点就在这里。

## 最小示例

```rust
fn pop_front(&mut self) -> Option<i32> {
    self.head.take().map(|old_head| {
        let next = old_head.borrow_mut().next.take();
        // 更新新 head 的 prev 或清空 tail
    })
}
```

## 注意点

- 双向链表每一步修改都可能牵动多处状态
- `borrow()` 和 `borrow_mut()` 的作用域要尽量短
- 遍历时通常会 clone Rc 指针，而不是移动节点本体

## 常见错误

- 只改了 head，忘了改新 head 的 prev
- 只处理多节点情况，忘了单节点和空链表
- 借用作用域过长，导致 RefCell 运行时冲突

## 我的理解

这一节让我更确定：双向结构的难点不是链表，而是“一次改动要维持多少不变量”。
