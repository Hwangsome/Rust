# 1. 代码组织

> 类型：**Study note**
> 关键词：package、crate、module
> 下一篇：[2. 模块基础](./2-模块基础.md)

## 一分钟结论

- `package`、`crate`、`module` 是三层不同概念
- `package` 由 `Cargo.toml` 描述
- `crate` 是编译单元
- `module` 是 crate 内部的命名空间和组织方式


## 定义

| 概念 | 它是什么 |
| --- | --- |
| package | Cargo 管理的项目单位 |
| crate | Rust 编译器实际编译的单元 |
| module | crate 内部的代码组织与命名空间机制 |

## 作用

- package 解决项目分发和依赖管理
- crate 解决编译边界
- module 解决内部结构和访问路径

## 原理

### 1. package 是最外层

你通常会通过 `Cargo.toml` 先认识一个 package。

### 2. crate 是编译视角

一个 package 里可以有：

- binary crate
- library crate

### 3. module 是 crate 内进一步拆分

它主要解决：

- 命名空间
- 可见性
- 职责拆分

## 最小示例

```text
package
  -> crate
      -> modules
```

## 注意点

### 1. 不要把这三个词混着用

初学者最容易说成“一个 module 就是一个 crate”，这是不准确的。

### 2. 组织结构是工程决策，不只是语法问题

你怎么拆 crate 和 module，会直接影响 API 边界和维护成本。

## 常见错误

### ❌ 错误 1：把 package、crate、module 当同义词

### ❌ 错误 2：只按文件夹理解 module，而不看命名空间语义

### ❌ 错误 3：项目一复杂就只会“继续往 main.rs 里塞”

## 我的理解

- package、crate、module 是 Rust 组织代码的三层骨架
- 这三层一旦分清，后面学模块和可见性就不会一直词不达意
