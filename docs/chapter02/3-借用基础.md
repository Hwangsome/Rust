## 这一节解决什么问题

上一篇讲到：用 `&T` 传参可以避免 move，调用方仍保留所有权。但这引出了一个新问题：

**如果多个地方都持有引用，一边读一边写，会发生什么？**

这正是多线程并发中数据竞争（data race）的经典场景。在 Rust 里，这类问题被提前到**编译期**解决：借用规则保证了"同一时刻，同一数据，要么多个读者，要么一个写者，二者不能同时存在"。

这一节把借用规则讲清楚，包括：
- `&T`（不可变引用）可以有多个
- `&mut T`（可变引用）必须独占
- NLL（Non-Lexical Lifetimes）：借用活到最后一次使用，不是到花括号结束
- 借用与 slice

---

## 一分钟结论

- `&T` 是**不可变借用**（shared reference）：可以同时有多个，不能修改值
- `&mut T` 是**可变借用**（mutable reference）：同一时刻只能有一个，可以修改值
- **借用规则**（核心）：
  - 多个 `&T` 可以并存
  - **或者**一个 `&mut T`（独占）
  - 二者**不能同时存在**
- **NLL**：借用的有效区间到"最后一次使用"为止，不是到词法作用域结束
- **切片**（`&[T]`、`&str`）是对某段数据的借用视图，也遵循借用规则

## 与其他语言对比

| 场景 | C++ | Java | Rust |
|-----|-----|------|------|
| 多个读者同时存在 | 允许（但程序员保证不写）| 允许（对象共享）| 允许（多个 `&T`）|
| 一个写者同时读 | 允许（可能产生 data race）| 允许（可能产生 data race）| **编译期禁止** |
| 多个写者同时存在 | 允许（未定义行为）| 允许（可能产生 data race）| **编译期禁止** |
| 保证机制 | 无（依赖程序员）| 无（依赖程序员）/ `synchronized` | 编译器强制 |

Java 里，你可以有任意多个对象引用，读写也不受限制——你需要自己用 `synchronized` 来防止数据竞争。Rust 把这个保证提前到了编译期。

## 核心概念与心智模型

### 借用规则的直觉：读者-写者问题

```
多读者 = 安全
  ┌──────────┐  ┌──────────┐  ┌──────────┐
  │  读者 A  │  │  读者 B  │  │  读者 C  │
  │   &T     │  │   &T     │  │   &T     │
  └─────┬────┘  └─────┬────┘  └─────┬────┘
        │             │             │
        └─────────────┴─────────────┘
                      ↓
              ┌──────────────┐
              │    数据      │  ← 大家一起看，互不干扰
              └──────────────┘


独占写者 = 安全
              ┌──────────────┐
              │   写者 W     │
              │   &mut T     │
              └──────┬───────┘
                     │
              ┌──────────────┐
              │    数据      │  ← 只有 W 在访问，不会有冲突
              └──────────────┘


读者+写者同时 = 危险！
  ┌──────────┐
  │  读者 A  │   ← 正在读
  │   &T     │
  └─────┬────┘
        │     ┌──────────────┐
        ├────→│    数据      │←──┐
        │     └──────────────┘  │
              ┌──────────┐      │
              │  写者 W  │──────┘  ← 同时在写
              │  &mut T  │         ← 读者 A 看到的值随时可能变！
              └──────────┘
Rust 在编译期禁止这种情况 → E0502
```

### 借用的"有效区间"（NLL 的核心）

**在 Rust 2018 Edition（NLL）之前**，借用的有效区间是词法作用域（花括号）：

```rust
// 老行为（没有 NLL）：
let mut v = vec![1, 2, 3];
let r = &v[0];  // 借用开始
println!("{r}");
// 这里 r 不再被使用，但在老版本里借用直到花括号才结束
v.push(4); // ❌ 老版本：即使 r 不再用，编译器也认为借用还活着
```

**NLL 之后（现代 Rust）**：借用活到最后一次使用为止：

```rust
// 现代 Rust（NLL）：
let mut v = vec![1, 2, 3];
let r = &v[0];
println!("{r}"); // ← r 最后一次使用，借用在这里结束
v.push(4); // ✅ r 的借用已经结束，可以 push
```

## 详细原理

### 1. 多个不可变借用可以并存

```rust
let values = vec![4, 5, 6];

// 可以同时有多个只读借用
let r1 = &values;
let r2 = &values;
let r3 = &values;

// 三个都可以用
println!("r1={r1:?}, r2={r2:?}, r3={r3:?}");
```

### 2. 可变借用独占

```rust
let mut values = vec![4, 5, 6];

{
    let r_mut = &mut values;  // 独占可变借用开始
    r_mut.push(7);
    println!("r_mut = {r_mut:?}");
    // r_mut 在这里的最后一次使用
}
// ↑ 离开块，r_mut 的借用结束

// 现在可以再次借用
let r_read = &values;
println!("r_read = {r_read:?}");
```

### 3. 为什么不能同时持有 `&T` 和 `&mut T`

```rust
let mut v = vec![1, 2, 3];
let r = &v;         // 不可变借用

// v.push(4);       // ❌ E0502：不能在 r 仍然活跃时修改 v
// let r2 = &mut v; // ❌ E0502：不能在 r 仍然活跃时创建可变借用

println!("{r:?}");  // r 的最后一次使用

// r 已经不再使用，借用结束，下面可以修改
v.push(4);          // ✅ NLL 允许这样
```

### 4. NLL 的实际效果

```rust
let mut data = vec![1, 2, 3];

// 情景 1：不可变借用在 push 前已经用完
let first = &data[0];
println!("first = {first}"); // NLL：first 的借用在这里结束
data.push(4);                 // ✅ 合法（first 的借用已结束）

// 情景 2：不可变借用在 push 后还在用
let first = &data[0];
data.push(5); // ❌ E0502：first 的借用还没结束（下面还要用 first）
println!("first = {first}"); // first 在这里才结束
```

### 5. 切片借用：对原数据的"视图"

```rust
let v = vec![10, 20, 30, 40, 50];

// 切片是借用，不拷贝数据
let whole: &[i32] = &v[..];    // 全部
let head: &[i32] = &v[..3];    // 前三个 [10, 20, 30]
let tail: &[i32] = &v[3..];    // 后两个 [40, 50]
let mid: &[i32] = &v[1..4];    // 中间 [20, 30, 40]

println!("whole = {whole:?}");
println!("head  = {head:?}");
println!("tail  = {tail:?}");
println!("mid   = {mid:?}");

// v 仍然是唯一所有者
println!("v 仍然有效: {v:?}");
```

## 完整运行示例

```rust
fn demonstrate_shared_borrows() {
    println!("=== 多个 &T 可以并存 ===");
    let text = String::from("hello, rust");

    let r1 = &text;
    let r2 = &text;
    let r3 = &text;

    // 三个都可以用
    println!("r1 = {r1}");
    println!("r2 = {r2}");
    println!("r3 长度 = {}", r3.len());
    // r1, r2, r3 的借用在此结束（NLL）

    // 现在可以 move text
    let owned = text; // ✅ 因为 r1/r2/r3 的借用已结束
    println!("owned = {owned}");
}

fn demonstrate_exclusive_mutable_borrow() {
    println!("\n=== &mut T 必须独占 ===");
    let mut numbers = vec![1, 2, 3];

    // 只有一个 &mut 在活跃
    let r_mut = &mut numbers;
    r_mut.push(4);
    println!("push 后 = {r_mut:?}");
    // r_mut 在此处最后一次使用，借用结束

    // 现在可以再次借用（不管是 & 还是 &mut）
    let r_read = &numbers;
    println!("read = {r_read:?}");

    numbers.push(5); // ✅ r_read 的借用也结束了
    println!("最终 = {numbers:?}");
}

fn demonstrate_nll() {
    println!("\n=== NLL：借用活到最后一次使用 ===");
    let mut data = vec![10, 20, 30];

    let first = &data[0];
    println!("first = {first}"); // NLL：first 的借用到这里结束

    data.push(40); // ✅ first 的借用已结束
    println!("data 现在 = {data:?}");
}

fn demonstrate_slice_borrow() {
    println!("\n=== 切片借用：零拷贝视图 ===");
    let v = vec![10, 20, 30, 40, 50];

    let head = &v[..2];
    let tail = &v[3..];

    println!("head = {head:?}");
    println!("tail = {tail:?}");

    // head 和 tail 都是 v 的视图，不拷贝数据
    println!("原 Vec v = {v:?}（仍然有效）");

    // 切片的实际应用：处理部分数据
    let sum_of_head: i32 = head.iter().sum();
    let sum_of_tail: i32 = tail.iter().sum();
    println!("head 之和 = {sum_of_head}, tail 之和 = {sum_of_tail}");
}

fn main() {
    demonstrate_shared_borrows();
    demonstrate_exclusive_mutable_borrow();
    demonstrate_nll();
    demonstrate_slice_borrow();
}
```

**预期输出**：

```text
=== 多个 &T 可以并存 ===
r1 = hello, rust
r2 = hello, rust
r3 长度 = 11
owned = hello, rust

=== &mut T 必须独占 ===
push 后 = [1, 2, 3, 4]
read = [1, 2, 3, 4]
最终 = [1, 2, 3, 4, 5]

=== NLL：借用活到最后一次使用 ===
first = 10
data 现在 = [10, 20, 30, 40]

=== 切片借用：零拷贝视图 ===
head = [10, 20]
tail = [40, 50]
原 Vec v = [10, 20, 30, 40, 50]（仍然有效）
head 之和 = 30, tail 之和 = 90
```

## 编译器错误分析

### ❌ E0502：同时持有 `&T` 和 `&mut T`

```rust
let mut v = vec![1, 2, 3];
let r = &v;       // 不可变借用
v.push(4);        // ❌ 尝试修改，但 r 还在活跃
println!("{r:?}"); // r 在这里用
```

```text
error[E0502]: cannot borrow `v` as mutable because it is also borrowed as immutable
  |
2 | let r = &v;
  |          - immutable borrow occurs here
3 | v.push(4);
  |   ^^^^^^^^ mutable borrow occurs here
4 | println!("{r:?}");
  |            - immutable borrow later used here
```

**修复**：把 `r` 的最后一次使用移到 `push` 之前：

```rust
let mut v = vec![1, 2, 3];
let r = &v;
println!("{r:?}"); // ← 先用完 r，借用结束

v.push(4);         // ✅ 此时 r 的借用已结束
```

### ❌ E0499：同时有两个 `&mut T`

```rust
let mut v = vec![1, 2, 3];
let r1 = &mut v;
let r2 = &mut v; // ❌
println!("{r1:?}");
```

```text
error[E0499]: cannot borrow `v` as mutable more than once at a time
  |
2 | let r1 = &mut v;
  |               - first mutable borrow occurs here
3 | let r2 = &mut v;
  |               ^ second mutable borrow occurs here
4 | println!("{r1:?}");
  |            -- first borrow later used here
```

**修复**：确保 r1 的最后一次使用在 r2 创建之前：

```rust
let mut v = vec![1, 2, 3];
let r1 = &mut v;
r1.push(10);        // r1 的最后一次使用（NLL 结束 r1 的借用）

let r2 = &mut v;    // ✅ r1 已结束
r2.push(20);
println!("{v:?}");
```

## 实际工程场景

### 1. 只读访问（函数只需要读时，用 `&T`）

```rust
fn find_duplicates(data: &[i32]) -> Vec<i32> {
    let mut seen = std::collections::HashSet::new();
    data.iter()
        .filter(|&&x| !seen.insert(x))
        .copied()
        .collect()
}

let nums = vec![1, 2, 3, 2, 4, 3, 5];
let dups = find_duplicates(&nums);
println!("重复: {dups:?}");  // [2, 3]
println!("原数据: {nums:?}"); // ✅ 仍然有效
```

### 2. 原地修改（需要改时，用 `&mut T`）

```rust
fn normalize(scores: &mut Vec<f64>) {
    if scores.is_empty() { return; }
    let max = scores.iter().cloned().fold(f64::NEG_INFINITY, f64::max);
    for s in scores.iter_mut() {
        *s /= max;
    }
}

let mut scores = vec![80.0, 100.0, 60.0];
normalize(&mut scores);
println!("归一化后: {scores:?}"); // [0.8, 1.0, 0.6]
```

### 3. 缓存/懒加载（需要在"看起来不可变"的 API 下修改内部状态）

```rust
use std::cell::Cell;

struct Counter {
    count: Cell<u64>, // 内部可变性（后续章节详讲）
}

impl Counter {
    fn new() -> Self { Counter { count: Cell::new(0) } }
    fn increment(&self) { self.count.set(self.count.get() + 1); } // &self，但能修改
    fn get(&self) -> u64 { self.count.get() }
}
```

## 性能影响

借用（引用）只是指针，8 或 16 字节（DST 的情况）：

- `&T`：8 字节（细指针）或 16 字节（胖指针）
- `&mut T`：同上

传引用始终比 move 整个 struct 要便宜（如果 struct 比指针大的话）：

```rust
struct BigStruct { data: [u8; 1024] } // 1KB

fn process_by_value(s: BigStruct) { /* 1KB 被复制到栈... */ }
fn process_by_ref(s: &BigStruct)  { /* 只传了 8 字节指针 */ }
```

但对 `i32`、`f64` 等小类型，传引用不一定比传值快（额外的间接访问可能更慢）——这类类型通常是 Copy，按值传即可。

## 注意点与陷阱

### 陷阱 1：循环里不能同时持有 `&T` 和 `&mut T`

```rust
let mut v = vec![1, 2, 3];
for r in &v {
    v.push(*r * 2); // ❌ 在 for 循环（不可变借用）里尝试 push（可变借用）
}
// 修复：收集要添加的元素，循环外再 push
let additions: Vec<i32> = v.iter().map(|&x| x * 2).collect();
v.extend(additions); // ✅
```

### 陷阱 2：RefCell 和 `borrow_mut()` 是运行时检查

```rust
use std::cell::RefCell;
let v = RefCell::new(vec![1, 2, 3]);
let r1 = v.borrow();     // 运行时不可变借用
let r2 = v.borrow_mut(); // ❌ 运行时 panic！不是编译错误！
// 这是内部可变性的代价：规则仍在，只是检查点推到了运行时
```

## 我的理解与记忆方法

**核心规则的读者-写者比喻**：

> 想象一本正在被多人协作的书：
> - 任意多个人可以**同时翻阅**（多个 `&T`）
> - 只有**一个人在修改**时，其他人都不能翻阅或修改（独占 `&mut T`）
> - 翻阅中的人离开了（最后一次使用结束），其他人才能开始修改

**NLL 的判断方法**：

> 找到这个引用"最后在哪里被使用"，那一行之后借用就结束了，和花括号在哪里无关。

**选择 `&T` 还是 `&mut T` 的决策**：

```
我需要修改值吗？
  是 → &mut T
  否 → &T（更宽泛，更安全）
```
