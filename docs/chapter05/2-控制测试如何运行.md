# 2. 控制测试如何运行

> 类型：**Study note**
> 关键词：`cargo test`、filter、`#[ignore]`、`#[should_panic]`
> 上一篇：[1. 单元测试](./1-单元测试.md)
> 下一篇：[3. 集成测试](./3-集成测试.md)

## 一分钟结论

- `cargo test` 不只是“全部跑一遍”
- 你可以按名字筛选、单独跑 ignored tests、查看测试输出
- `#[should_panic]` 用来表达“panic 才是预期”

## 证据来源

- 对应模块：[topic_02_controlling_how_tests_are_run.rs](../../chapters/chapter05/src/topic_02_controlling_how_tests_are_run.rs)
- 实际运行：`cargo test -p chapter05 -- --nocapture`

关键结果：

```text
test controlling_how_tests_are_run::tests::ignored_demo ... ignored
test controlling_how_tests_are_run::tests::named_filter_demo ... ok
test controlling_how_tests_are_run::tests::should_panic_demo - should panic ... ok
```

## 定义

测试控制指的是：在不改测试代码主体的情况下，调整**哪些测试运行、怎样运行、输出如何显示**。

## 作用

- 提高日常调试效率
- 区分快速测试和慢测试
- 更精准地观察特定测试行为

## 原理

当前章节给出的典型命令：

```bash
cargo test -p chapter05
cargo test -p chapter05 add_returns_sum
cargo test -p chapter05 -- --ignored
cargo test -p chapter05 -- --show-output
```

对应测试属性：

- `#[ignore]`：默认不跑
- `#[should_panic]`：预期 panic

## 最小示例

```rust
#[test]
#[ignore]
fn ignored_demo() {
    assert!(true);
}

#[test]
#[should_panic(expected = "radius should be positive")]
fn should_panic_demo() {
    panic!("radius should be positive");
}
```

## 注意点

### 1. `ignored` 不等于废弃

它常用于慢测试或环境敏感测试。

### 2. `should_panic` 是在测试“失败也是预期”

这很适合验证边界保护逻辑。

### 3. 筛选测试时，名字本身很重要

测试名写得清楚，过滤体验才会好。

## 常见错误

### ❌ 错误 1：只会 `cargo test` 全量跑

### ❌ 错误 2：慢测试、脆弱测试和快测试不分层

### ❌ 错误 3：看到 panic 就默认测试失败

在 `#[should_panic]` 场景下，panic 恰恰是成功条件。

## 我的理解

- 测试运行控制本质上是在做“反馈回路优化”
- 跑得快、定位准，比机械全量跑更重要
