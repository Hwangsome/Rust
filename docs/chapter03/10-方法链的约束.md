# 10. 方法链的约束

> 类型：**Study note**
> 关键词：method chaining、builder、`Result`
> 上一篇：[9. 引用的转换与赋值](./9-引用的转换与赋值.md)

## 一分钟结论

- 方法链不是“看起来顺就能一直链下去”
- 它依赖每一步返回值的类型仍然支持下一步调用
- 一旦某一步返回 `Result`、`Option` 或别的包装类型，链的形状就会改变

- 对应模块：[topic_10_method_chaining_constraints.rs](../../chapters/chapter03/src/topic_10_method_chaining_constraints.rs)
- 运行章节：`cargo run -p chapter03`


## 扩展演示输出（当前代码已升级）

`topic_10_method_chaining_constraints.rs` 演示 4 个链条变化：
1. 每步返回 `Self`，链可以无限延续（`append` / `uppercase`）
2. 某步返回 `Result<Self, E>`，链的**形状变了**——下一步不能直接 `.append(...)`
3. 恢复 A：外层函数返回 `Result`，用 **`?`** 把 `Result` 拆回 `Self`，继续链
4. 恢复 B：用 `map` / `and_then` 在 `Result` 内部继续操作

```text
-- (1) 只要每步返回 Self，链可以无限延续 --
simple = TextBuilder { value: "HELLO, WORLD" }

-- (3) 恢复 A：用 ? 展开 Result，继续链 --
build_bounded() = Ok(TextBuilder { value: "Rust Language is safe" })

-- (4) 恢复 B：用组合子 map / and_then --
using_map = Ok(TextBuilder { value: "OK!" })
will_fail = Err("length 24 exceeds 10")
```

## 定义

方法链是把多个方法调用连续串起来的写法：

```rust
value.method1().method2().method3()
```

## 作用

- 减少中间变量
- 更贴近“流水线式”思维
- 在 builder 风格 API 中很常见

## 原理

### 1. 链能继续，前提是返回类型对得上

当前示例：

```rust
TextBuilder::new().append("Rust").append(" Lang")
```

之所以能继续，是因为 `append` 返回的仍然是 `Self`。

### 2. 返回 `Result` 后，链的主体就变了

```rust
.try_limit(20)
```

一旦这一步返回 `Result<Self, String>`：

- 当前值不再是 `TextBuilder`
- 而是 `Result<TextBuilder, String>`

后续能不能继续调用，取决于 `Result` 自己提供什么方法。

### 3. builder 风格常用“按值拿走 self 再返回 self”

```rust
fn append(mut self, text: &str) -> Self
```

这种写法让链式 API 很自然，但本质上仍然在走所有权转移。

## 最小示例

```rust
#[derive(Debug)]
struct TextBuilder {
    value: String,
}

impl TextBuilder {
    fn new() -> Self {
        Self { value: String::new() }
    }

    fn append(mut self, text: &str) -> Self {
        self.value.push_str(text);
        self
    }
}
```

## 注意点

### 1. 链式调用不是语法糖万能药

一旦返回类型变复杂，链也会变得更难读。

### 2. `self` 按值传递意味着对象在链中持续被移动

这不是坏事，但一定要意识到背后的所有权行为。

### 3. `Result` / `Option` 会改变 API 的可调用面

你不能假设“前面链得上，后面就还能链”。

## 常见错误

### ❌ 错误 1：只看调用形式，不看返回类型

这会让方法链的断点看起来像“突然发生”。

### ❌ 错误 2：以为 builder 风格没有所有权成本

其实它经常正是建立在按值接收 `self` 上的。

### ❌ 错误 3：链太长时还拒绝拆中间变量

有些链拆开反而更容易理解和排错。

## 我的理解

- 方法链的本质不是“连续点号”
- 而是“每一步的返回类型刚好能接上下一步”
- 真正理解这一点后，很多 fluent API 就不再神秘
