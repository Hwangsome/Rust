# 5. 使用外部依赖

## 一分钟结论

- 第三方依赖通常通过 `Cargo.toml` 的 `[dependencies]` 引入
- `cargo add <crate-name>` 是最常见入口
- 真正难点不在“怎么加”，而在“该不该加、加哪个、版本怎么控”

## 定义

外部依赖就是当前 crate 之外的 crate，通过 Cargo 进行声明和解析。

## 作用

- 复用生态能力
- 减少重复造轮子
- 更快接入已有协议、格式、工具链

## 原理

最常见流程：

1. 在 `Cargo.toml` 里声明依赖
2. Cargo 下载并解析版本
3. 编译时把依赖纳入 crate 图
4. 代码中通过 `use` 引入需要的项

## 最小示例

```toml
[dependencies]
serde = "1"
```

```bash
cargo add serde
```

## 注意点

### 1. 能自己写五行，不代表一定不该依赖

但也不要因为“有 crate”就立刻引入。要权衡：

- 维护状态
- 文档质量
- 版本稳定性
- 二进制体积

### 2. 依赖是长期承诺

一旦引入，就要承担升级、兼容性和安全更新成本。

### 3. `use` 只是把名字带进作用域，不是安装依赖

安装发生在 Cargo 层，不在 `use` 语句里。

## 常见错误

### ❌ 错误 1：把 `use` 当成“引入外部包”

### ❌ 错误 2：只看功能，不看维护质量

### ❌ 错误 3：依赖一多就不再审视 API 面

依赖不仅影响功能，也影响编译时间、体积和升级复杂度。

## 我的理解

- 依赖管理是一种架构决策，不只是 Cargo 命令
- 真正成熟的做法，是把“为什么引入这个 crate”讲清楚
