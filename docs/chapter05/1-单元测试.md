# 1. 单元测试

> 类型：**Study note**
> 关键词：unit test、`#[test]`、`#[cfg(test)]`
> 下一篇：[2. 控制测试如何运行](./2-控制测试如何运行.md)

## 一分钟结论

- 单元测试通常和被测代码放在同一个文件里
- `#[cfg(test)]` 让测试代码只在测试阶段编译
- `#[test]` 标记某个函数为测试入口

## 证据来源

- 对应模块：[unit_testing.rs](../../chapters/chapter05/src/unit_testing.rs)
- 运行测试：`cargo test -p chapter05`

当前被测函数：

```rust
pub fn add(left: i32, right: i32) -> i32 {
    left + right
}
```

对应测试：

```rust
#[test]
fn add_returns_sum() {
    assert_eq!(add(2, 3), 5);
}
```

## 定义

单元测试是针对一个较小行为单元的验证，通常离实现代码最近。

## 作用

- 验证局部行为是否符合预期
- 让实现和验证靠得更近
- 便于快速定位回归问题

## 原理

### 1. `#[cfg(test)]` 控制编译时机

测试模块只在 `cargo test` 时参与编译。

### 2. `#[test]` 标记测试函数

测试函数通常：

- 无参数
- 无返回值
- 内部包含断言

## 最小示例

```rust
pub fn add(left: i32, right: i32) -> i32 {
    left + right
}

#[cfg(test)]
mod tests {
    use super::add;

    #[test]
    fn add_returns_sum() {
        assert_eq!(add(2, 3), 5);
    }
}
```

## 注意点

### 1. 单元测试强调“贴近实现”

这也是它通常放在同文件里的原因。

### 2. 断言比打印更重要

打印能帮助观察，断言才真正定义通过条件。

## 常见错误

### ❌ 错误 1：把单元测试写成演示代码，没有断言

### ❌ 错误 2：测试和实现隔太远，导致维护成本高

### ❌ 错误 3：只测试“看起来最正常”的路径

## 我的理解

- 单元测试最有价值的地方，是让“实现”和“预期”放在一起
- 它不是替代所有测试，而是最细粒度的一层保护网
