# 3. 统一多种错误类型

## 定义

当一个函数里会遇到多种不同错误来源，而你又想连续使用 `?` 时，就需要把这些错误统一到一种返回类型里。

## 作用

- 让一个函数保持单一 `Result<T, E>` 签名
- 保留不同错误来源的信息
- 让高层调用者还能按错误种类分支处理

## 原理

常见做法是定义一个错误枚举，然后在调用点用 `map_err` 把底层错误映射到统一错误类型。

## 最小示例

```rust
let temp_celsius =
    temperature_from_sensor(sensor_id).map_err(TemperatureError::Sensor)?;
```

## 注意点

- `?` 要求最终错误类型兼容
- 统一错误类型不等于丢失错误来源
- 如果项目里这种情况很多，后面可以考虑 `thiserror`

## 常见错误

- 一遇到多种错误就直接 `unwrap`
- 统一错误时只保留字符串，失去结构化信息
- 忽视“调用者是否还需要分辨错误类别”

## 我的理解

这节让我更明确：错误类型设计本身就是 API 设计的一部分，不只是“为编译器服务”。
