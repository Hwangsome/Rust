# 3. 集成测试

> 类型：**Study note**
> 关键词：integration test、`tests/`、smoke
> 上一篇：[2. 控制测试如何运行](./2-控制测试如何运行.md)

## 一分钟结论

- 集成测试通常放在 crate 根目录下的 `tests/`
- 它更像“从 crate 外部使用你的代码”
- 单元测试测局部实现，集成测试测整体接口

## 证据来源

- 对应模块：[integration_testing.rs](../../chapters/chapter05/src/integration_testing.rs)
- 集成测试文件：[tests/smoke.rs](../../chapters/chapter05/tests/smoke.rs)
- 实际运行：`cargo test -p chapter05 -- --nocapture`

关键结果：

```text
Running tests/smoke.rs
test smoke_test ... ok
```

## 定义

集成测试是站在 crate 外部视角，对整体行为做验证的测试。

## 作用

- 验证公开接口能否协同工作
- 模拟真实使用方式
- 补足单元测试只看局部实现的盲区

## 原理

当前项目给出的最小示例在：

```text
chapters/chapter05/tests/smoke.rs
```

内容非常小：

```rust
#[test]
fn smoke_test() {
    assert_eq!(2 + 2, 4);
}
```

虽然简单，但它已经说明了目录和运行模型。

## 最小示例

```rust
// tests/smoke.rs
#[test]
fn smoke_test() {
    assert_eq!(2 + 2, 4);
}
```

## 注意点

### 1. 集成测试更关注“整体接口”

它不一定要知道每个内部函数怎么实现。

### 2. 目录位置有语义

放在 `tests/` 下，运行方式和普通模块内测试不同。

### 3. smoke test 是很常见的最小集成测试形态

它不追求覆盖一切，而是先确认系统最基本路径没坏。

## 常见错误

### ❌ 错误 1：把集成测试写成另一份单元测试

### ❌ 错误 2：只写非常重的端到端测试，不写最小 smoke 测试

### ❌ 错误 3：把所有内部实现细节都暴露给集成测试

## 我的理解

- 集成测试真正重要的不是“文件放哪”，而是“视角变了”
- 它在问：如果我是 crate 使用者，这套东西能不能正常工作
