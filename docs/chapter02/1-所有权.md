# 1. 所有权基础

> 类型：**Study note**
> 关键词：ownership、move、clone、Copy、heap
> 下一篇：[2. 函数中的所有权](./2-函数中的所有权.md)

## 一分钟结论

- Rust 里每个值都有**所有者**
- 同一时刻通常只有一个所有者
- `String` 这类堆分配值赋值后默认会 **move**
- `i32` 这类 `Copy` 类型赋值后原值仍可继续使用
- `clone()` 是显式深拷贝，不是“顺手复制一下”

## 定义

所有权是 Rust 管理内存和资源生命周期的核心规则。最小版本可以记成三条：

1. 每个值都有一个所有者
2. 同一时刻只能有一个所有者
3. 所有者离开作用域时，值会被清理

## 作用

所有权机制的目标是：

- 不依赖 GC 也能安全管理内存
- 避免双重释放
- 避免悬垂引用
- 让资源释放时机更清楚

## 原理

### 1. 不是所有赋值都是复制

```rust
let s1 = String::from("world");
let s2 = s1;
```

这不是“得到两个完全独立的 `String`”。默认语义是：

- `s1` 的所有权移动给 `s2`
- `s1` 之后不再有效

### 2. `clone()` 是显式复制

```rust
let s1 = String::from("world");
let s2 = s1.clone();
```

这里表示：

- 保留 `s1`
- 新建一个独立拷贝给 `s2`

### 3. `Copy` 类型赋值更像传统直觉里的复制

```rust
let x = 15;
let y = x;
```

对于 `i32` 这类 `Copy` 类型：

- `x` 赋给 `y` 后
- `x` 仍然可用

这也是为什么新手容易误判：他们把所有类型都想成 `Copy` 了。

## 最小示例

```rust
fn main() {
    let s1 = String::from("world");
    let s2 = s1.clone();
    println!("s1 = {}, s2 = {}", s1, s2);

    let x = 15;
    let y = x;
    println!("x = {}, y = {}", x, y);
}
```

## 注意点

### 1. `clone` 不是免费操作

它通常意味着额外分配和复制成本，不应该当成“消错按钮”。

### 2. move 不是 bug，它是默认资源转移语义

尤其是对持有堆数据的类型，这样的设计能避免释放责任不清。

### 3. `Copy` 是一类特殊语义，不是所有类型天然都有

不要把 `i32` 的直觉直接套到 `String`、`Vec<T>` 上。

## 常见错误

### ❌ 错误 1：move 后继续使用原值

```rust
let s1 = String::from("hello");
let s2 = s1;
println!("{} {}", s1, s2);
```

错误信息：

```text
borrow of moved value
```

### ❌ 错误 2：一遇到 move 就无脑 `clone`

这能让代码暂时编译，但不一定表达了正确的资源语义。

### ❌ 错误 3：把所有赋值都当复制

这正是 Rust 初学阶段最常见的认知偏差。

## 我的理解

- 所有权不是“Rust 的怪规则”，而是它整套内存安全模型的根
- move 不是编译器刁难你，而是在明确“谁负责这个值”
- `clone`、`Copy`、move 三者必须分清，否则后面借用会一直混
