# 9. 引用的转换与赋值

> 类型：**Study note**
> 关键词：reborrow、reference assignment、`&mut`、`&`
> 上一篇：[8. 解构的 Struct 参数](./8-解构的Struct参数.md)

## 一分钟结论

- 可变引用有时可以临时借出一个只读视图
- 这类现象常被叫做 **reborrow**
- 不可变引用赋值给另一个变量时，行为更像复制引用
- 真正要盯住的是：当前哪种借用还活着

## 证据来源

- 对应模块：[topic_09_casting_and_assignment_of_references.rs](../../chapters/chapter03/src/topic_09_casting_and_assignment_of_references.rs)

## 定义

这一节关注的是引用之间的关系，而不是引用和值的关系：

- 可变引用能否借出不可变视图
- 不可变引用赋值时是什么语义

## 作用

- 帮你理解更细粒度的借用行为
- 为后面读复杂 API 和链式方法调用打基础
- 减少“为什么这里突然又能借 / 又不能借”的困惑

## 原理

### 1. 可变引用可以临时 reborrow 成不可变视图

```rust
let mutable_ref = &mut data;
let immutable_view: &i32 = &*mutable_ref;
```

这表示：

- 先有一个可变引用
- 再从它暂时借出一个只读视图

### 2. 不可变引用赋值更像“复制引用”

```rust
let ref_text_1 = &text;
let ref_text_2 = ref_text_1;
```

这不会拿走底层 `text` 的所有权，它只是让两个变量都指向同一个只读引用视图。

## 最小示例

```rust
fn main() {
    let mut data = 42;
    let mutable_ref = &mut data;

    let immutable_view: &i32 = &*mutable_ref;
    println!("{}", immutable_view);

    *mutable_ref = 43;
    println!("{}", *mutable_ref);
}
```

## 注意点

### 1. 关键不是“语法能不能写”，而是借用区间有没有结束

reborrow 是否安全，本质上仍然取决于借用规则。

### 2. 不可变引用“像 Copy”，不等于所有引用都能随便转

尤其是 `&mut` 相关行为，要更谨慎看待。

### 3. 引用赋值不等于复制底层数据

它复制的是访问路径，而不是拥有的值。

## 常见错误

### ❌ 错误 1：把引用赋值当成底层值复制

两者语义完全不同。

### ❌ 错误 2：看到 reborrow 成功，就以为借用规则可以随便绕

Rust 只是允许受约束的临时视图，不是放弃规则。

### ❌ 错误 3：忘了区分“引用本身”和“引用后面的值”

这会让很多看似简单的赋值和传参都变得混乱。

## 我的理解

- 这一节像是借用模型里的“精修部分”
- 真正关键的是建立：引用本身也有层次和生命周期，而不是一个模糊指针概念
