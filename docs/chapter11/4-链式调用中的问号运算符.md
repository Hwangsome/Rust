## 扩展演示输出（当前代码已升级）

`topic_04_chaining_question_marks.rs` 文件头说明如何把 chapter03 学过的"非错误版 builder"升级为带校验版本：把每一步的返回类型从 `Self` 换成 `Result<Self, E>`，外层函数也返回 `Result`，用 `?` 保持链条通顺。

## 定义

这一节讨论的是：当方法本身支持链式调用时，`?` 如何融入其中。

## 作用

- 提高多步可失败流程的可读性
- 让“验证 -> 支付 -> 发货”之类的流程更像业务语义
- 避免一层层嵌套 match

## 原理

只要每一步都返回兼容的 `Result<&mut Self, E>` 或类似形状，链式调用就能继续；一旦某一步失败，`?` 会立刻提前返回。

## 最小示例

```rust
self.process_payment()?.ship_order()?;
```

## 注意点

- 链得起来的前提仍然是返回类型兼容
- `validate()?;` 和后续 `&mut self` 方法之间要注意借用作用域
- 可读性很重要，链太长时不要拒绝拆开

## 常见错误

- 只看“能不能链”，不看类型怎么变化
- 以为加了 `?` 以后错误就“处理完了”
- 返回类型不一致时还硬写成一条链

## 我的理解

这节让我更清楚：方法链的本质永远是“上一步输出能不能成为下一步输入”，`?` 只是把失败分支提前 return 掉。
