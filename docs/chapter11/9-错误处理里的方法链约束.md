## 定义

这一节回到方法链本身，但重点不是 fluent API，而是“错误返回类型会怎样改变链的形状”。

## 作用

- 帮你判断什么时候还能继续点下去
- 理解为什么错误处理会影响 API 设计
- 给 builder、流程对象、状态机式 API 提供判断依据

## 原理

返回 `&mut Self` 的方法还能继续接需要 `&mut self` 或 `&self` 的方法；一旦返回 `Result<&mut Self, E>`，当前链的主体就变成 `Result`，后续需要 `match`、`?` 或适配器接手。

## 最小示例

```rust
match account.try_min_balance(10_000) {
    Ok(account) => println!("{}", account.balance),
    Err(error) => println!("{}", error),
}
```

## 注意点

- 方法链的根本前提永远是返回类型兼容
- 错误类型不是附属品，它会决定 API 还能不能顺着链
- 链很长时要考虑调试和阅读成本

## 常见错误

- 只从语法角度理解方法链
- 一返回 `Result` 就不知道为什么后面点不下去了
- 为了链式外观硬压扁错误处理逻辑

## 我的理解

这一节像是在提醒我：错误处理从来不是“后面补一下”，它会直接反过来塑造 API 的样子。
