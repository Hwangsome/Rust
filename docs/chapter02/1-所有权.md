## 这一节解决什么问题

几乎所有其他编程语言都要求程序员手动管理内存（C/C++）或者依赖垃圾回收（Java/Python/Go）。这两种方式各有代价：

- 手动管理：双重释放（double free）、悬垂指针（dangling pointer）、内存泄漏
- 垃圾回收：运行时开销、GC 暂停、不可预测的内存释放时机

Rust 选择了第三条路：**所有权系统**。编译器在编译期分析每一块内存的归属，精确决定何时释放，**不需要 GC，也不需要手动 free**。

这一节是理解 Rust 的起点。后续所有内容（借用、生命周期、智能指针、并发安全）都建立在这三条规则上：

1. 每个值有且只有一个所有者（owner）
2. 同一时刻只能有一个所有者
3. 所有者离开作用域（scope）时，值被自动释放（drop）

---

## 一分钟结论

- Rust 里的"赋值"**默认是 move（移动）**，不是复制
- `String`、`Vec<T>`、`Box<T>` 等堆分配类型赋值后，原变量**失效**（E0382）
- `i32`、`bool`、`char`、`f64` 等原生类型实现了 `Copy`，赋值后原变量**仍可用**
- `clone()` 是**显式深拷贝**，会分配新内存，不是"修复 E0382 的快捷键"
- 值在**所有者离开花括号（作用域）时自动 drop**——无需手动 free
- **Shadowing（遮蔽）**不等于 move：它是创建同名新绑定，把旧绑定遮住

## 与其他语言对比

| 操作 | C | C++ | Java | Python | Rust |
|-----|---|-----|------|--------|------|
| `a = b`（String） | 浅拷贝（指针赋值） | 深拷贝（`operator=`） | 引用赋值（两个引用指向同一对象）| 引用赋值 | **Move**（b 失效） |
| `a = b`（int） | 复制值 | 复制值 | 复制值（int 是 primitive） | 对象复制（小整数缓存）| 复制值（`Copy`） |
| 内存释放 | 手动 `free` | 析构函数 | GC | GC / 引用计数 | **自动 drop（作用域结束）** |
| 同一内存两次释放 | 未定义行为（崩溃）| 析构两次（崩溃）| 不可能（GC）| 不可能（引用计数）| **编译期禁止**（E0382）|

## 核心概念与心智模型

### 栈（Stack）vs 堆（Heap）

理解所有权，先要理解 Rust 如何使用内存：

```
┌─────────────────────────────────────────────────────────┐
│                     栈（Stack）                          │
│  快速分配/释放，大小固定，随函数调用自动管理              │
│                                                          │
│  let x: i32 = 42;                                       │
│  ┌──────────┐                                           │
│  │    42    │  ← 4 字节，编译期已知，直接在栈上         │
│  └──────────┘                                           │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│              栈 + 堆（Heap）的组合                        │
│                                                          │
│  let s: String = String::from("hello");                 │
│                                                          │
│  栈上（24 字节）：                                        │
│  ┌──────────┬──────────┬──────────┐                     │
│  │ ptr      │ len (5)  │ cap (5)  │                     │
│  └──────────┴──────────┴──────────┘                     │
│       │                                                  │
│       ▼ 指向堆                                           │
│  ┌─────────────┐                                         │
│  │ h e l l o  │  ← 5 字节实际内容（堆上）               │
│  └─────────────┘                                         │
└─────────────────────────────────────────────────────────┘
```

`i32` 完全在栈上，大小固定，可以直接复制（Copy 语义）。`String` 在栈上保存"指针+长度+容量"，实际字节在堆上，需要 move 语义来保证"同一时刻只有一个所有者"。

### Move 的本质

```rust
let s1 = String::from("hello");
let s2 = s1;  // ← 这一行发生了什么？
```

```
赋值前：
  s1: ┌──────────┬──────────┬──────────┐
      │ ptr→堆   │ len = 5  │ cap = 5  │
      └──────────┴──────────┴──────────┘
               │
               ▼ 堆
          ┌─────────────┐
          │ h e l l o  │
          └─────────────┘

赋值后（move）：
  s1: 失效（编译器标记为已 moved，不能再用）

  s2: ┌──────────┬──────────┬──────────┐
      │ ptr→堆   │ len = 5  │ cap = 5  │
      └──────────┴──────────┴──────────┘
               │
               ▼ 同一块堆内存（s2 接管了所有权）
          ┌─────────────┐
          │ h e l l o  │
          └─────────────┘
```

Move 不是复制内存，它是所有权的转移：栈上的结构体字段被按位复制到 s2，但 s1 被标记为"已移出"，编译器拒绝你再使用它。这样堆内存只会被释放一次（当 s2 离开作用域时）。

### Copy vs Move

```
实现了 Copy 的类型（栈上完整数据，无堆资源）：
  i8/16/32/64/128/isize, u8/16/32/64/128/usize
  f32/64
  bool
  char
  元组（所有成员都是 Copy）
  数组（元素类型是 Copy）
  引用 &T（不管 T 是否 Copy）

  赋值 = 按位复制，原变量仍可用

没有 Copy 的类型（持有堆资源或需要精细管理所有权）：
  String, Vec<T>, Box<T>, HashMap, ...

  赋值 = Move（所有权转移，原变量失效）
```

## 详细原理

### 1. Copy 类型：赋值后原变量仍可用

```rust
let x = 42_i32;
let y = x;   // i32 是 Copy，按位复制栈上的值

// x 和 y 各自有一份独立的 42
println!("x = {x}"); // ✅ 仍然有效
println!("y = {y}"); // ✅ 仍然有效

// 同样适用于 tuple 和 array（成员都是 Copy 时）
let pair = (1_i32, true);
let pair2 = pair;
println!("pair = {:?}", pair);   // ✅
println!("pair2 = {:?}", pair2); // ✅
```

### 2. String 的 Move：原变量失效

```rust
let s1 = String::from("hello");
let s2 = s1;  // s1 的所有权 move 给 s2

// println!("{}", s1); // ❌ E0382: 不能用已被 move 的值
println!("{}", s2);    // ✅ s2 是新的所有者
```

**为什么不能"两个都用"？**

如果 s1 和 s2 都"有效"，当它们都离开作用域时，同一块堆内存会被 drop 两次——这是双重释放 bug，会导致内存损坏或崩溃。Rust 在编译期就阻止了这种情况。

### 3. `clone()`：显式深拷贝

```rust
let s1 = String::from("hello");
let s2 = s1.clone();  // 分配新内存，复制堆上的内容

println!("s1 = {s1}"); // ✅ s1 仍然有效
println!("s2 = {s2}"); // ✅ s2 是独立的副本

// s1 和 s2 是完全独立的 String，互不影响
let mut s2 = s2;
s2.push_str(" world");
println!("s1 = {s1}"); // hello（未受影响）
println!("s2 = {s2}"); // hello world
```

> **注意**：`clone()` 不是"遇到 E0382 就用"的快捷键。它有堆分配和内存复制的成本。如果你只是需要读数据，借用（`&T`）是更好的选择。

### 4. 作用域与 Drop

```rust
fn scope_demo() {
    let s = String::from("hello");  // s 从这里开始有效

    {
        let inner = String::from("world");  // inner 从这里开始有效
        println!("{inner}");
    }  // ← inner 离开作用域，被 drop（堆内存释放）

    println!("{s}");  // ✅ s 仍然有效
}  // ← s 离开作用域，被 drop
```

Rust 在每个 `}` 处自动插入 drop 调用，按照"后声明的先 drop"（LIFO）的顺序。这是**确定性析构**——你确切知道每个值何时被释放。

### 5. Shadowing 与 Move 的区别

```rust
// Shadowing（遮蔽）：创建同名新绑定
let s = String::from("hello");
let s = s.to_uppercase();  // 旧 s 被 drop，新 s 是新的 String
println!("{s}");  // "HELLO"

// 对 Copy 类型的 Shadowing：可以改类型
let x = 5;
let x = x + 1;        // 仍是 i32，值变了
let x = format!("{x}");  // 类型从 i32 变成 String
println!("{x}");  // "6"
```

Shadowing 和 `mut` 是不同的：
- `mut`：同一个绑定可以改变值，**类型不能变**
- Shadowing：创建新绑定，旧绑定不再可见，**类型可以变**

## 完整运行示例

```rust
use std::mem::size_of;

fn demonstrate_stack_vs_heap() {
    println!("=== 栈 vs 堆的内存大小 ===");
    println!("i32 栈大小:    {} bytes", size_of::<i32>());
    println!("String 栈大小: {} bytes (ptr+len+cap)", size_of::<String>());
    println!("&str 栈大小:   {} bytes (胖指针:ptr+len)", size_of::<&str>());
    println!();
}

fn demonstrate_copy_semantics() {
    println!("=== Copy 类型：赋值后原变量仍可用 ===");

    let x = 42_i32;
    let y = x;
    println!("x = {x}, y = {y}  ← 两个都有效");

    let pair = (1_i32, true, 3.14_f64);
    let pair2 = pair;
    println!("pair = {pair:?}, pair2 = {pair2:?}  ← tuple 也是 Copy");
    println!();
}

fn demonstrate_move_semantics() {
    println!("=== Move 类型：赋值后原变量失效 ===");

    let s1 = String::from("hello");
    let s2 = s1;
    // println!("{s1}"); // ← 取消注释会 E0382
    println!("s2 = {s2}  ← s1 已被 move 给 s2，s1 失效");

    let v1 = vec![1, 2, 3];
    let v2 = v1;
    // println!("{v1:?}"); // ← 取消注释会 E0382
    println!("v2 = {v2:?}  ← v1 已被 move 给 v2");
    println!();
}

fn demonstrate_clone() {
    println!("=== clone()：显式深拷贝 ===");

    let s1 = String::from("hello");
    let mut s2 = s1.clone();  // 新内存，新副本

    s2.push_str(" world");
    println!("s1 = {s1}  ← 未受影响");
    println!("s2 = {s2}");
    println!();
}

fn demonstrate_scope_and_drop() {
    println!("=== 作用域与自动 Drop ===");

    let outer = String::from("outer");
    println!("outer 创建: {outer}");

    {
        let inner = String::from("inner");
        println!("inner 创建: {inner}");
        // inner 在这里 drop（释放堆内存）
    }
    println!("inner 已被 drop（离开了花括号）");
    println!("outer 仍然有效: {outer}");
    // outer 在这里 drop
}

fn demonstrate_shadowing() {
    println!("\n=== Shadowing vs mut ===");

    // mut：修改同一绑定的值（类型不能变）
    let mut count = 0;
    count += 1;
    count += 1;
    println!("mut count = {count}");

    // Shadowing：创建新绑定（类型可以变）
    let spaces = "   ";          // &str
    let spaces = spaces.len();   // usize（类型变了！）
    let spaces = format!("len={spaces}"); // String（再次变）
    println!("shadowed spaces = {spaces}");
}

fn main() {
    demonstrate_stack_vs_heap();
    demonstrate_copy_semantics();
    demonstrate_move_semantics();
    demonstrate_clone();
    demonstrate_scope_and_drop();
    demonstrate_shadowing();
}
```

**预期输出**：

```text
=== 栈 vs 堆的内存大小 ===
i32 栈大小:    4 bytes
String 栈大小: 24 bytes (ptr+len+cap)
&str 栈大小:   16 bytes (胖指针:ptr+len)

=== Copy 类型：赋值后原变量仍可用 ===
x = 42, y = 42  ← 两个都有效
pair = (1, true, 3.14), pair2 = (1, true, 3.14)  ← tuple 也是 Copy

=== Move 类型：赋值后原变量失效 ===
s2 = hello  ← s1 已被 move 给 s2，s1 失效
v2 = [1, 2, 3]  ← v1 已被 move 给 v2

=== clone()：显式深拷贝 ===
s1 = hello  ← 未受影响
s2 = hello world

=== 作用域与自动 Drop ===
outer 创建: outer
inner 创建: inner
inner 已被 drop（离开了花括号）
outer 仍然有效: outer

=== Shadowing vs mut ===
mut count = 2
shadowed spaces = len=3
```

## 编译器错误分析

### ❌ E0382：使用已被 move 的值

```rust
let s1 = String::from("hello");
let s2 = s1;
println!("{s1}"); // ❌
```

```text
error[E0382]: borrow of moved value: `s1`
 --> src/main.rs:3:15
  |
1 | let s1 = String::from("hello");
  |     -- move occurs because `s1` has type `String`,
  |        which does not implement the `Copy` trait
2 | let s2 = s1;
  |          -- value moved here
3 | println!("{s1}");
  |           ^^ value borrowed here after move
  |
  = help: consider cloning the value if the usage is appropriate
```

**修复方案（根据实际需求选择）**：

```rust
// 方案 A：如果不需要两个独立值，只用 s2
let s2 = s1;
println!("{s2}");

// 方案 B：如果需要两个独立副本，clone
let s2 = s1.clone();
println!("{s1}");  // 原值仍可用
println!("{s2}");  // 独立副本

// 方案 C：如果只是想读，用借用
let s2 = &s1;      // 借用，不 move
println!("{s1}");  // 仍然有效
println!("{s2}");  // 通过引用访问
```

### ❌ E0384：修改不可变变量

```rust
let x = 5;
x = 6; // ❌
```

```text
error[E0384]: cannot assign twice to immutable variable `x`
  |
1 | let x = 5;
  |     - first assignment to `x`
2 | x = 6;
  |     ^
  |
help: consider making this binding mutable
  |
1 | let mut x = 5;
  |     +++
```

**修复**：`let mut x = 5;`

## 实际工程场景

### 1. 函数参数中的 Move

```rust
fn consume(s: String) {
    println!("消费了: {s}");
    // s 在函数结束时被 drop
}

fn main() {
    let s = String::from("hello");
    consume(s);
    // println!("{s}"); // ❌ s 已经被 move 进 consume 了

    // 解决：如果 consume 只需要读，应该改签名
}

// 更好的设计：不 move，只借用
fn just_read(s: &str) {
    println!("读取: {s}");
}

fn main_v2() {
    let s = String::from("hello");
    just_read(&s);    // 传引用
    println!("{s}"); // ✅ s 仍然有效
}
```

### 2. 在 struct 字段间 move

```rust
struct Config {
    host: String,
    port: u16,
}

let cfg = Config { host: "localhost".into(), port: 8080 };

// 取出 host 字段（部分 move）
let host = cfg.host;    // host 字段被 move
// println!("{}", cfg.host); // ❌ cfg.host 已被 move
println!("{}", cfg.port);   // ✅ port 是 Copy，仍可用
// println!("{cfg:?}"); // ❌ cfg 部分被 move，不能整体使用
```

### 3. `Drop` trait——自定义释放行为

```rust
struct Resource {
    name: String,
}

impl Drop for Resource {
    fn drop(&mut self) {
        println!("释放资源: {}", self.name);
    }
}

fn main() {
    let r1 = Resource { name: "DB连接".into() };
    {
        let r2 = Resource { name: "文件句柄".into() };
        println!("使用 r2");
    }  // ← "释放资源: 文件句柄"
    println!("r1 仍然有效");
}  // ← "释放资源: DB连接"
```

这是 RAII（Resource Acquisition Is Initialization）模式——资源的生命周期和 struct 的生命周期绑定，不可能忘记释放。

## 性能影响

### Move 是零成本的

Move 只是把栈上的字节（通常 8~24 字节）复制到新位置，然后标记旧位置为无效。**堆上的数据不会被复制**。

```rust
let s1 = String::from("包含大量数据的超长字符串...");
let s2 = s1; // 只复制了 24 字节的栈数据，堆数据原地不动
```

### Clone 有成本

```rust
let s1 = String::from("1GB 的字符串");
let s2 = s1.clone(); // 分配新内存 + 复制 1GB 数据——成本很高！
```

### Copy 是按位复制（便宜）

```rust
let x = 42_i32;
let y = x; // 复制 4 字节，和 int 赋值一样快
```

## 注意点与陷阱

### 陷阱 1：函数调用也是 Move

```rust
fn eat(s: String) { ... }  // 按值接收，会 move
let s = String::from("hello");
eat(s);           // s 被 move 进函数
// println!("{s}"); // ❌ E0382
```

### 陷阱 2：String 和 &str 的赋值行为不同

```rust
let s1: String = String::from("hello");
let s2 = s1;  // move

let s3: &str = "world";  // &str 是 Copy（它只是一个胖指针）
let s4 = s3;  // copy，s3 仍然有效
```

### 陷阱 3：`clone()` 不应该是"遇到错误就加"

```rust
// ❌ 无脑 clone
fn bad(names: Vec<String>) -> Vec<String> {
    let cloned = names.clone(); // 复制了整个 Vec，可能很贵
    do_something(names);
    cloned
}

// ✅ 思考是否真的需要两份数据
fn better(names: Vec<String>) {
    for name in &names {  // 借用，不 clone
        println!("{name}");
    }
}
```

## 我的理解与记忆方法

**核心比喻**：

> Rust 的 Move 就像**转移一张"所有权证书"**：有证书的那个变量才能访问值。赋值就是把证书交给别人，你自己的那份证书就作废了。

> Copy 就像**直接复印**：两份都是真的，互不影响。

**判断某个类型是否 Copy 的规则**：
- 类型是否完全在栈上（无堆分配）？是 → 大概率是 Copy
- 有没有 `impl Drop for T`？有 → 一定**不是** Copy（不允许同时存在）
- 是否明确标注了 `#[derive(Copy)]`？是 → 是 Copy

**E0382 的正确处理顺序**：
1. 是否需要两份独立数据？→ `clone()`
2. 是否只需要读？→ 改用 `&T` 借用
3. 是否只需要用一次？→ 直接 move，不要那个"原值"
