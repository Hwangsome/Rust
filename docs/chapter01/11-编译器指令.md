# 11. 编译器指令

## 一分钟结论

- Rust 的“编译器指令”通常就是 **attribute**
- 写法主要有两种：`#[...]` 和 `#![...]`
- `#[...]` 常作用于某个项
- `#![...]` 常作用于整个文件或模块
- 它们不是运行时逻辑，而是在告诉编译器“怎么处理这段代码”

## 定义

Rust attribute 是附着在代码上的一层“编译期元信息”，常见用途包括：

- 开关某类 lint
- 控制测试编译
- 自动派生 trait
- 条件编译

## 作用

它们的作用不是“增加业务功能”，而是：

- 控制编译器行为
- 调整告警和错误策略
- 给类型或模块附加额外语义
- 让一些重复样板由编译器代劳

## 原理

### 1. 外层 attribute：`#[...]`

```rust
#[allow(dead_code)]
fn square(x: i32) -> i32 {
    x * x
}
```

含义：

- 这个 attribute 只作用于当前函数
- 它告诉编译器：即使这个函数没被调用，也不要报 `dead_code`

### 2. 内层 attribute：`#![...]`

```rust
#![allow(unused_variables)]
```

含义：

- 它通常放在文件顶部
- 作用范围更大，常覆盖整个 crate 或当前模块

### 3. 它们主要是“编译期控制”，不是运行期特性

也就是说：

- 你不会在程序运行时“执行”一个 attribute
- 它更像是在编译前告诉工具链怎么理解这段代码

## 最小示例

```rust
#![allow(unused_variables)]

#[allow(dead_code)]
fn square(x: i32) -> i32 {
    x * x
}

fn main() {
    let message = String::from("attributes use #[] syntax");
    println!("{message}");
}
```

## 注意点

### 1. `#[...]` 和 `#![...]` 作用范围不同

- `#[...]` 更像“给下面这个项加说明”
- `#![...]` 更像“给整个文件 / 模块加说明”

### 2. `allow` 不是默认答案

如果只是为了让代码安静，就到处写 `allow`，最后会把真正有价值的告警也埋掉。

### 3. attribute 很常见，但不要把它们当魔法

例如你以后会经常见到：

- `#[derive(Debug)]`
- `#[cfg(test)]`
- `#[test]`

它们都属于同一类机制，只是用途不同。

## 常见错误

### ❌ 错误 1：把 attribute 当成运行时代码

它不会像函数那样“执行”，它更像编译器配置和附加说明。

### ❌ 错误 2：分不清 `#[...]` 和 `#![...]`

如果范围理解错了，就很容易出现“为什么这个规则只影响了一行 / 为什么整个文件都受影响”的困惑。

### ❌ 错误 3：遇到告警先 `allow`

正确顺序通常应该是：

1. 先理解告警在说什么
2. 再判断它是不是合理
3. 最后才决定是否压制

## 我的理解

- attribute 是 Rust 里非常核心的一层“编译期沟通语言”
- 它让代码不仅能表达逻辑，也能表达“该怎么被编译器处理”
- 真正重要的不是记住每个 attribute，而是先理解这套机制
